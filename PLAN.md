# PLAN.md — 关联账号查询性能优化实施计划

> **目标**：在不改变当前“关联账号查询”业务语义的前提下，优先压缩后台查询链路中的 N+1 查询、无效补查和高基数候选带来的响应时间抖动，稳定管理后台在大候选集场景下的可用性。
>
> **范围**：仅优化 `association_query` 相关读取路径、候选补充信息加载方式、缓存策略与必要的查询辅助函数；不改动当前相似度计算规则、不重写现有账号关联判定逻辑。
>
> **原则**：先消灭最重的 N+1 和无效查询，再做批量化与懒加载，最后评估是否继续做结果物化；每一步都必须保持 SQLite / MySQL / PostgreSQL 三库兼容。

---

## 1. 当前现状与主要瓶颈

### 1.1 关联查询主链路存在明显 N+1
当前 `service/association_query.go` 在遍历候选用户时，会逐个执行：

- `getUserBrief(candidateUID)`
- `model.GetLinkByUsers(targetUserID, candidateUID)`
- `GetSharedIPs(targetUserID, candidateUID)`（按需）

这意味着候选人数一多，数据库查询次数会线性膨胀。

### 1.2 补充信息加载时机过早
当前逻辑在候选用户尚未完成：

- 相似度筛选
- 最小置信度过滤
- 排序
- `limit` 截断

之前，就为候选人加载用户信息、既有关联状态与共享 IP。这会导致大量“最终不会返回给前端”的候选用户也触发补查。

### 1.3 `account_links` 存在逐 pair 存在性查询
当前 `model/account_link.go` 中的 `FindExistingLink()` 会对每个 pair 执行一次：

```sql
SELECT * FROM account_links WHERE user_id_a = ? AND user_id_b = ? LIMIT 1
```

这类查询“查不到”本身是常态，但现在会形成：

- 大量单条 SQL
- 大量 `record not found` 日志噪音
- 整体查询耗时被小查询堆高

### 1.4 `users` 表也存在逐个补查
当前 `getUserBrief()` 会对每个候选用户执行一次单独的 `users` 查询。候选集大时，这也是标准 N+1。

### 1.5 候选用户指纹加载也存在线性扇出
当前每个 candidate 都会单独调用：

- `GetLatestFingerprints(candidateUID, 10)`
- 无流水时再调 `GetDeviceProfilesAsFingerprints(candidateUID)`

这部分虽然这次日志里不是最先炸出来的热点，但随着候选集扩大，也会成为下一层瓶颈。

---

## 2. 优化目标与验收指标

## 2.1 总体目标

- 将关联查询从“候选数线性放大 SQL 数量”调整为“候选粗筛 + 最终结果批量补齐”
- 明显降低 `account_links` 与 `users` 的单条重复查询次数
- 将后台关联页的主要时间消耗集中到真正有价值的候选上
- 减少 `record not found` 噪音日志，提升问题定位清晰度

## 2.2 阶段验收指标

| 指标 | P0 目标 | P1 目标 | P2 目标 |
|------|---------|---------|---------|
| 关联查询总 SQL 数量 | 下降 50%+ | 下降 70%+ | 长期稳定 |
| 后台关联页 p95 延迟 | 下降 30%+ | 下降 45%+ | 接近缓存命中延迟 |
| `account_links` 单次查询条数 | 从逐条变为批量 | 保持稳定 | 保持稳定 |
| `users` 补查次数 | 从逐条变为批量 | 保持稳定 | 保持稳定 |
| 三库兼容性 | 无回归 | 无回归 | 无回归 |

---

## 3. 分阶段总览

| 阶段 | 目标 | 是否改 schema | 预计周期 |
|------|------|---------------|----------|
| **P0** | 先消灭最明显的 N+1 和早补查 | 否 | 1~3 天 |
| **P1** | 批量化 candidate 指纹加载与共享信息补查 | 否 | 3~5 天 |
| **P2** | 评估是否做快照/物化结果 | 可选 | 视数据规模而定 |

---

## 4. P0：先做最小风险的查询链路收敛

## 4.1 P0 目标

优先砍掉最重的数据库扇出，不动现有关联算法，只调整“什么时候查、怎么批量查”。

### Task P0-1：将候选补充信息加载延后到排序/过滤之后

**是否改 schema**：否

#### 实施内容

- [ ] 将 `service/association_query.go` 里的主循环拆成“粗算阶段”和“补齐阶段”
- [ ] 第一阶段只保留：
  - `candidateUID`
  - `confidence`
  - `tier`
  - `explanation`
  - `matchedDimensions`
  - `matchDimensions` / `totalDimensions`
- [ ] 完成排序、`min_confidence` 过滤和 `limit` 截断后
- [ ] 只对最终结果集补充：
  - `User`
  - `ExistingLink`
  - `SharedIPs`
  - `Details`（如果当前接口仍需要）

#### 涉及文件

- `service/association_query.go`

#### 预期收益

- 大幅减少无意义补查
- 把数据库成本集中在真正返回给前端的结果上

#### 风险

- 需要小心保持返回结构与前端兼容

#### 验收标准

- [ ] 最终返回结果内容不变
- [ ] 候选较多时，SQL 数量明显下降
- [ ] 排序与过滤逻辑无回归

---

### Task P0-2：批量加载既有关联状态，替代逐 pair 查询

**是否改 schema**：否

#### 实施内容

- [ ] 在 `model/account_link.go` 增加批量查询函数
- [ ] 输入：目标用户 ID + 最终候选用户 ID 列表
- [ ] 一次性查询这些 pair 对应的 `account_links`
- [ ] 在内存中按规范化 pair（`user_id_a:user_id_b`）构建 map
- [ ] `association_query` 从 map 中填充 `ExistingLink`
- [ ] 避免继续对每个 candidate 调 `GetLinkByUsers()`

#### 涉及文件

- `model/account_link.go`
- `service/association_query.go`
- `model/account_link_test.go`（如已有）
- `service/association_query_test.go`

#### 预期收益

- 把 N 次 `account_links` 查询收敛成 1 次
- 消灭大量 `record not found` 噪音

#### 风险

- 需要确保 pair 规范化逻辑与现有唯一键语义一致

#### 验收标准

- [ ] 最终结果中的 `ExistingLink` 与原行为一致
- [ ] `account_links` 查询从逐条变为批量
- [ ] 日志中不再充满逐条 `record not found`

---

### Task P0-3：批量加载用户简要信息，替代逐个 `getUserBrief`

**是否改 schema**：否

#### 实施内容

- [ ] 增加批量加载用户摘要的辅助函数
- [ ] 通过 `WHERE id IN ?` 一次性拉取最终结果集涉及的用户
- [ ] 在内存中构建 `map[int]UserBrief`
- [ ] 对查不到或已软删的用户保留兜底：`UserBrief{ID: userID}`
- [ ] 将 `getUserBrief()` 保留给单用户场景，不再用于候选批量循环

#### 涉及文件

- `service/association_query.go`
- 可能涉及 `model/user.go` 或相关用户查询文件
- `service/association_query_test.go`

#### 预期收益

- 把 N 次 `users` 查询收敛成 1 次
- 降低软删/不存在用户导致的噪音查询

#### 风险

- 需要确认用户表软删条件与现有行为一致

#### 验收标准

- [ ] 候选结果中的用户信息与原行为一致
- [ ] `users` 查询从逐个变为批量
- [ ] 不存在用户时返回兜底结构而非异常失败

---

### Task P0-4：将共享 IP 补查限制在最终结果集

**是否改 schema**：否

#### 实施内容

- [ ] 保持 `includeSharedIPs` 语义不变
- [ ] 但只在最终过滤后的结果集上执行共享 IP 查询
- [ ] 不再对全部候选用户执行共享 IP 补查
- [ ] 如成本仍高，后续再考虑拆详情接口或继续批量化

#### 涉及文件

- `service/association_query.go`

#### 预期收益

- 减少高成本附加查询
- 降低“打开关联页即触发大量补查”的峰值开销

#### 风险

- 无明显业务风险，主要是实现顺序调整

#### 验收标准

- [ ] `includeSharedIPs=true` 时返回结果正确
- [ ] SQL 数量仅与最终返回条数相关，而不是候选总数相关

---

### Task P0-5：把“查不到是常态”的单条查询从错误噪音中剥离

**是否改 schema**：否

#### 实施内容

- [ ] 评估 `FindExistingLink()` 这类存在性查询是否继续保留
- [ ] 若保留单条查询路径，则避免用会产生 `record not found` 噪音的写法
- [ ] 对“未命中不是异常”的场景，使用更安静的查询方式
- [ ] 统一审视 `association_query` 里类似的“未命中非异常”路径

#### 涉及文件

- `model/account_link.go`
- `service/association_query.go`

#### 预期收益

- 降低误导性日志
- 让真实错误更容易被发现

#### 风险

- 需要避免把真正异常吞掉

#### 验收标准

- [ ] 常规未命中不再刷屏日志
- [ ] 真正的 SQL/连接错误仍能被保留

## 5. P1：继续向下压缩指纹候选加载成本

## 5.1 P1 目标

在 P0 消灭主 N+1 之后，进一步处理 candidate 指纹加载的线性扇出。

### Task P1-1：批量加载候选用户最新指纹

**是否改 schema**：否

#### 实施内容

- [ ] 评估为最终候选集批量加载 `user_fingerprints`
- [ ] 统一按 `user_id` 分组后在内存里截取每用户最近 N 条
- [ ] 流水为空时，再评估是否批量补设备档案兜底
- [ ] 保持现有相似度计算输入语义不变

#### 涉及文件

- `model/fingerprint.go`
- `service/association_query.go`
- `service/association_query_test.go`

#### 预期收益

- 再砍一层 candidate N+1
- 降低大候选集场景下的总体耗时

#### 风险

- 批量查回的总数据量可能增大，需要预算控制

#### 验收标准

- [ ] 查询次数下降
- [ ] 单次拉回数据量受控
- [ ] 相似度结果不发生明显回退

---

### Task P1-2：评估共享 IP 批量聚合或详情懒加载

**是否改 schema**：否

#### 实施内容

- [ ] 若 `SharedIPs` 仍是热点，评估做批量聚合
- [ ] 或将共享 IP 明细改为展开时懒加载
- [ ] 保持列表页优先轻量返回

#### 涉及文件

- `service/association_query.go`
- `controller/admin_fingerprint.go`
- 前端关联页相关文件

#### 预期收益

- 继续降低列表页首次加载压力

#### 风险

- 前端可能需要配合一个附加请求

#### 验收标准

- [ ] 共享 IP 不再成为列表页瓶颈
- [ ] 前端体验可接受

## 6. P2：如果数据继续增长，再评估结果物化

## 6.1 P2 目标

若 P0/P1 后后台关联页仍受实时计算影响，再考虑引入更重的结构性方案。

### Task P2-1：评估关联结果摘要缓存或物化快照

**是否改 schema**：可选

#### 实施内容

- [ ] 评估是否为用户维护 TopN 关联摘要缓存/快照
- [ ] 后台列表优先读快照，实时重算作为兜底
- [ ] 快照仅保存列表需要的轻量字段

#### 涉及文件

- `service/association_query.go`
- `service/cron_fingerprint.go`
- `model/account_link.go`
- 可能新增快照模型/缓存逻辑

#### 预期收益

- 后台列表进一步从“现算”转向“读取现成结果”

#### 风险

- 结果存在短时间滞后
- 实现与维护成本更高

#### 验收标准

- [ ] 常见后台列表不再依赖全量实时计算
- [ ] 刷新逻辑可控、可回滚

## 7. 测试与发布要求

## 7.1 功能正确性

- [ ] 关联结果排序、置信度、命中维度不变
- [ ] `ExistingLink` 展示语义不变
- [ ] 用户不存在/软删时仍有兜底展示
- [ ] `includeSharedIPs` 开关行为不变

## 7.2 性能验证

- [ ] 对比优化前后的 SQL 数量
- [ ] 对比后台关联页接口 p50 / p95 / p99
- [ ] 对比候选集较大时的响应时间
- [ ] 记录批量查询前后数据库日志变化

## 7.3 三库兼容

- [ ] SQLite 验证 `IN` 批量查询、排序与过滤正确性
- [ ] MySQL 验证批量 pair 查询与性能表现
- [ ] PostgreSQL 验证批量查询与现有索引路径兼容

## 7.4 发布策略

- [ ] 优先以最小改动上线 P0
- [ ] 每个任务保持独立可回滚
- [ ] 先观测后台查询耗时与 SQL 数量，再推进 P1/P2

## 8. 实施顺序建议

### 第一轮

- [ ] 完成 P0-1：延后补充信息加载
- [ ] 完成 P0-2：批量查 `account_links`
- [ ] 完成 P0-3：批量查 `users`

### 第二轮

- [ ] 完成 P0-4：共享 IP 仅对最终结果补查
- [ ] 完成 P0-5：清理无意义 `record not found` 噪音

### 第三轮

- [ ] 评估并推进 P1-1：批量加载候选指纹
- [ ] 如仍有必要，再推进 P1-2 / P2

---

## 9. 结论

这次优化最核心的不是重写关联算法，而是把当前“边遍历候选边逐个补查”的链路，改成：

1. **先粗算候选结果**
2. **再排序、过滤、截断**
3. **最后对最终结果做批量补齐**

只要 P0 做扎实，当前后台“关联账号查询”这条链路的绝大部分时间浪费都会先被压下去；之后再视数据规模决定是否继续做 candidate 指纹批量化或结果物化，不提前过度设计。
